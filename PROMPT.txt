缓存代理后端服务。

背景：给上层业务和多个异构存储后端（内存缓存、Redis、远端 HTTP 服务、远端 gRPC 服务）之间架一个统一的 KV 中间层，负责按规则路由请求到合适的后端、本地缓存加速、保证多副本间数据一致性，让业务方不用关心数据到底落在哪里、是不是最新的、有没有被换到。

需求：多后端注册要支持四种类型——memory（进程内 LRU/TTL）、redis（远端 Redis 实例）、http（REST 风格的 KV 服务）、grpc（gRPC KV 服务），每种后端要有权重配置（请求按权重轮询或加权随机分发）；后端实例可以多个同类型并列（比如两台 Redis 一主一从），按 key 前缀规则决定命中哪个。键前缀匹配规则要支持多种策略：精确前缀（如 `user:profile:*` 走 Redis）、通配符（如 `cache:*`）、哈希路由（key 哈希取模落到固定后端）、黑白名单（禁用某些前缀走特定后端）。

需求：读写模式要支持三种——Cache-Aside（业务先查缓存，没命中回源后端再回填，失败容忍）、Write-Through（写穿透，业务写时同步写后端，强一致慢一点）、Write-Behind（写异步批量回写后端，快但可能丢数据）。版本号被动一致性——每个 key 关联一个单调递增的版本号，业务回传版本号检查发现不一致就拒绝，避免读到过期数据。

需求：Pub-Sub 与 Gossip 主动失效——本地缓存失效时要主动通知其他节点清掉对应 key；Pub-Sub 走 Redis Pub/Sub 或 NATS 这种现成通道；Gossip 模式走 SWIM 协议做节点间状态同步，适合节点多的场景。健康检查要每个后端实例定时 ping（TCP 探活或协议级 ping），连续失败 N 次自动剔除（不进路由表），连续成功 M 次自动恢复；剔除/恢复的事件要有日志和 metrics。

需求：Prometheus JSON 指标端点要暴露：每后端的 QPS、P50/P95/P99 延迟、命中率、剔除次数、错误码分布；格式走 Prometheus exposition 格式（`/metrics` 端点）。大 value（超过阈值比如 1MB）要透传不缓存——避免大 value 把内存缓存撑爆，透传走旁路直接打到后端。

需求：键前缀路由要支持动态路由——业务发请求时带上 routing key hint（不是真 key 而是分类标签），代理根据 hint 选后端，不依赖 key 内容。熔断可选配置——后端连续报错就熔断（直接返错不重试），熔断期间定时半开探测，恢复后再开。

需求：配置要走 viper + YAML，热加载（监听文件变更自动 reload 不重启服务）；日志走 zerolog 走 JSON 输出，方便 ELK 采集。错误响应要明确指出是后端错误还是路由错误（业务方排查问题用），不要一股脑返 500。

技术栈：Go + Gin HTTP 框架；ristretto 进程内缓存；fasthttp 做 HTTP 客户端（高性能）；gocron 做定时任务（健康检查）；viper + YAML 配置；zerolog 结构化日志；默认端口 8090。
